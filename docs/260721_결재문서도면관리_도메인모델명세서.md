## 1. 개요 및 도메인 설계 방향
안녕하세요, **PRODEV**입니다.
결재 통합 문서도면관리 시스템(DMS)을 구성하는 **8대 코어 도메인(Domain Aggregates & Entities)**의 도메인 추출 이유, 역할, 주요 속성(Fields), 그리고 각 도메인이 구동해야 하는 비즈니스 메서드(Methods)를 상세히 명시한 **결재 통합 문서도면관리 시스템 도메인 모델 상세 명세서(v1.0)**를 작성하여 보관합니다.

본 시스템의 모든 도메인 객체는 Domain-Driven Design(DDD) 원칙을 준수하여 **상태 변경과 비즈니스 검증 로직이 도메인 내부 메서드에 캡슐화**되도록 정밀 설계되었습니다.

## 2. 8대 코어 도메인 모델 상세 명세

### 2.1. Document (통합 문서/도면 마스터 - Aggregate Root)

**1. 구성 이유 및 역할**
  - **구성 이유:** 사내 일반 품의서 및 도면의 공통 마스터 정보를 통합 통제하고, 결재 진행 상태(`approvalStatus`), 가공 현장 유효성(`lifecycleStatus`), ABAC 접근 권한 속성을 Single Source of Truth(SSOT)로 유지하기 위함.
  - **역할:** 문서 생명주기 통제, 다단계 결재 전이 제어, 및 개정 타임라인 계보(`previousVersionDocumentId`) 1:1 역방향 단선 연결 총괄.

**2. 주요 속성 (Attributes)**
  - `id` (String / UUID): 문서 마스터 고유 식별자 (PK).
  - `documentNo` (String): 규칙 기반 문서/도면 번호.
  - `title` (String): 문서 및 도면 명칭.
  - `documentType` (DocumentType Enum): 문서 종류 (`REQ`, `DRAW`, `PUR`).
  - `workflowType` (WorkflowType Enum): 워크플로우 구분 (`INTERNAL` 자체기안, `EXTERNAL` 바이어 즉시승인).
  - `approvalStatus` (ApprovalStatus Enum): 결재 진행 상태 (`DRAFT`, `UNDER_REVIEW`, `APPROVED`, `REJECTED`, `CANCELLED`).
  - `lifecycleStatus` (LifecycleStatus Enum): 가공 현장 유효 등급 (`DRAFT`, `ACTIVE`, `SUPERSEDED`, `DEPRECATED`, `DELETED`).
  - `securityGrade` (SecurityGrade Enum): ABAC 보안 등급 (`GRADE_1`, `GRADE_2`, `GRADE_3`).
  - `ownerDepartment` (String): ABAC 소유 부서 코드.
  - `projectCode` (String): ABAC 수행 프로젝트 코드.
  - `versionMajor` (Integer): 대버전 번호.
  - `versionMinor` (Integer): 소버전 번호.
  - `previousVersionDocumentId` (String): 직전 승인 도면 부모 ID (Lineage 단선 연결 FK).
  - `versionLock` (Long): JPA 낙관적 락(@Version) 동시성 제어용 버전 번호.
  - `attachments` (List<DocumentAttachment>): 1:N 매핑된 기안 복수 첨부파일 리스트 (주 본문 파일 + 참고 증빙 첨부파일).

**3. 주요 비즈니스 메서드 (Methods)**
  - `addAttachment(DocumentAttachment attachment)`: 기안 작성 시 본문 도면/문서 파일 및 참고 증빙 자료(견적서, 시방서 등)를 복수로 첨부 추가합니다.
  - `removeAttachment(String attachmentId)`: 기안 임시저장(`DRAFT`) 상태에서 지정한 첨부파일을 제거합니다.
  - `submitApproval()`: 결재 상신을 수행하며, 필수 본문 첨부파일이 존재하고 무결성이 검증되었는지 확인 후 `UNDER_REVIEW`로 전이시킵니다.
  - `approve()`: 결재 최종 승인을 완료하며, `approvalStatus`를 `APPROVED`로 변경합니다. (자체 도면일 경우 `lifecycleStatus`를 `ACTIVE`로 격상)
  - `reject(String reason)`: 결재를 반려하며, `approvalStatus`를 `REJECTED`로 변경 후 사유를 기록합니다.
  - `cancel()`: 상신자가 기안을 취소하며, 상태를 `CANCELLED`로 전이시킵니다.
  - `markAsActive()`: 바이어 도면(`EXTERNAL`) 비동기 파일 검사 통과 시 `lifecycleStatus`를 `ACTIVE`로 격상시킵니다.
  - `supersede(String newDocId)`: 동일 품번의 신규 도면이 ACTIVE가 되었을 때 자신의 `lifecycleStatus`를 `SUPERSEDED`로 자동 전환합니다.
  - `softDelete()`: 문서를 논리 삭제 처리하여 `lifecycleStatus`를 `DELETED`로 변경합니다.
  - `restore()`: 관리자가 논리 삭제된 문서를 `ACTIVE` 상태로 원복 복원시킵니다.

### 2.2. Drawing (정밀 도면 메타데이터 및 물리 파일 - Entity)

**1. 구성 이유 및 역할**
  - **구성 이유:** 도면의 물리 파일 특성(품번, 기종, 체크섬, 파일 무결성 상태 `fileState`) 및 MariaDB 가상 컬럼(`active_part_key`) 기반 유니크 제약을 부모 Document와 1:1 식별자 공유로 관리하기 위함.
  - **역할:** 원본 도면 파일 및 뷰어용 변환 미디어(PDF/썸네일/OCR)의 무결성을 보장하고 품번-기종별 활성 도면 유일성을 물리 제어.

**2. 주요 속성 (Attributes)**
  - `drawingId` (String / UUID): 도면 고유 식별자 (PK이자 부모 Document.id 참조 FK).
  - `partNo` (String): 부품 번호 (Part Number).
  - `partName` (String): 부품 명칭.
  - `modelGroup` (String): 적용 기종 코드 (Model Group).
  - `drawingType` (DrawingType Enum): 도면 구분 (`DEV` 개발, `PROD` 양산).
  - `storageKey` (String): S3/NAS 물리 스토리지 보관 키.
  - `originalFileName` (String): 원본 파일명.
  - `fileSize` (Long): 바이트 단위 파일 크기.
  - `checksumSha256` (String): 파일 위변조 검증용 SHA256 암호화 해시값.
  - `fileState` (FileState Enum): 물리 파일 무결성 상태 (`STAGED`, `PENDING_CONFIRM`, `CONFIRMED`, `MOVE_FAILED`, `CORRUPTED`).
  - `docApprovalStatus` (ApprovalStatus Enum): 부모 문서 결재 상태.
  - `docLifecycleStatus` (LifecycleStatus Enum): 부모 문서 라이프사이클 상태.
  - `activePartKey` (String): `doc_lifecycle_status = 'ACTIVE'` 일 때만 `partNo_modelGroup`을 생성하는 MariaDB 가상 컬럼.

**3. 주요 비즈니스 메서드 (Methods)**
  - `confirmFileValidation()`: 비동기 바이러스/체크섬 검사 통과 시 `fileState`를 `CONFIRMED`로 변경하고 무결성 승인 이벤트를 발행합니다.
  - `markAsCorrupted(String reason)`: 검사 실패 시 `fileState`를 `CORRUPTED`로 마스킹하고 현장 노출을 강제 차단합니다.
  - `updateStorageKey(String newKey)`: Staging Area에서 Production Area로 파일 이관 성공 시 스토리지 키를 갱신합니다.

### 2.3. ApprovalLine & ApprovalAuditLog (전자 결재선 및 이력 스냅샷 - Value Object / Entity)

**1. 구성 이유 및 역할**
  - **구성 이유:** 결재선 차수별 심사자 정보와 승인/반려/대결 이력 스냅샷을 수정 불가능한 Append-Only 형태 원장으로 보존하기 위함.
  - **역할:** 다단계 승인 흐름 관제 및 법적 증적 확보, Side-by-Side JSON Diff 관제 제공.

**2. 주요 속성 (Attributes)**
  - `logId` (Long): 감사 로그 고유 식별자 (PK).
  - `documentId` (String): 소속 부모 문서 ID (FK).
  - `stepNo` (Integer): 결재 차수 번호 (1차, 2차, 3차 등).
  - `approverId` (String): 결재자/심사자 ID.
  - `actionType` (String): 행위 유형 (`SUBMIT`, `APPROVE`, `REJECT`, `PROXY_APPROVE`, `AUTO_APPROVE_EXTERNAL`).
  - `proxyApproverId` (String): 대결 수행 시 원 결재자 ID.
  - `comment` (String): 심사 의견.
  - `snapshotJson` (String): 결재 당시의 문서 메타데이터 및 서명 정보 JSON 스냅샷.
  - `loggedAt` (Timestamp): 결재 및 로그 작성 시각.

**3. 주요 비즈니스 메서드 (Methods)**
  - `createDecisionLog(String approverId, String action, String comment)`: 결재 권한자의 승인/반려 행위를 감사 스냅샷과 함께 영구 기록합니다.
  - `createProxyLog(String proxyId, String originalId, String reason)`: 대결 위임 승인 사유를 검증하고 로그를 남깁니다.

### 2.4. BOMItem (상하위 품목 조립 계층 - Entity)

**1. 구성 이유 및 역할**
  - **구성 이유:** 제품 조립체 도면과 하위 부품 도면 간의 계층적 부모-자식 관계(Bill of Materials)를 도메인 객체로 포용하여 현장 오가공 및 오적용을 방지하기 위함.
  - **역할:** 상하위 품목 간 1:N 조립 관계 매핑, 부품 수량 보관 및 N:M 연관 그래프 시각화 데이터 제공.

**2. 주요 속성 (Attributes)**
  - `bomItemId` (Long): BOM 항목 식별자 (PK).
  - `parentPartNo` (String): 상위 조립체 부품 번호 (Parent Part Number).
  - `childPartNo` (String): 하위 구성 부품 번호 (Child Part Number).
  - `quantity` (Integer): 소요 수량 (Quantity).
  - `unit` (String): 수량 단위 (EA, SET 등).
  - `isEssential` (Boolean): 필수 부품 여부 플래그.

**3. 주요 비즈니스 메서드 (Methods)**
  - `addChildPart(String childPartNo, Integer qty)`: 상위 품목에 하위 부품을 수량과 함께 신규 등록 매핑합니다.
  - `updateQuantity(Integer newQty)`: 부품 소요 수량을 변경 업데이트합니다.

### 2.5. SearchFilterPreset (사용자 맞춤 검색 필터 - Value Object / Entity)

**1. 구성 이유 및 역할**
  - **구성 이유:** 대용량 DMS 환경에서 사용자가 자주 사용하는 다차원 검색 필터 조건(기종, 기간, 보안등급 등)을 개인화 프리셋으로 고속 재사용하기 위함.
  - **역할:** DMS 탐색기(`/explorer`) 화면에서 사용자별 필터 즐겨찾기 보존.

**2. 주요 속성 (Attributes)**
  - `filterId` (Long): 저장된 필터 식별자 (PK).
  - `userId` (String): 소속 사용자 ID.
  - `filterName` (String): 사용자가 지정한 필터 프리셋 명칭.
  - `queryJson` (String): 다차원 필터링 조건 컬렉션 JSON 객체.
  - `searchedAt` (Timestamp): 검색 수행 시각.

**3. 주요 비즈니스 메서드 (Methods)**
  - `savePreset(String name, String json)`: 검색 필터 조건 프리셋을 즐겨찾기로 저장합니다.
  - `renamePreset(String newName)`: 즐겨찾는 필터 이름을 변경합니다.

### 2.6. SystemNotification (실시간 시스템 및 무결성 알림 - Entity)

**1. 구성 이유 및 역할**
  - **구성 이유:** 결재 수신 요청, 외부 도면 파일 검증 완료, 개정 알림을 수신자별 독립 DB 레코드로 적재 관리하기 위함.
  - **역할:** 비동기 도메인 이벤트 수신 시 알림 적재 및 실시간 SSE(Server-Sent Events) 브로드캐스팅.

**2. 주요 속성 (Attributes)**
  - `notificationId` (Long): 알림 고유 식별자 (PK).
  - `recipientId` (String): 알림 수신 대상 사용자 ID.
  - `title` (String): 알림 제목.
  - `content` (String): 알림 세부 내용.
  - `readStatus` (Boolean): 읽음/안읽음 상태 (`true`/`false`).
  - `createdAt` (Timestamp): 알림 발생 시각.

**3. 주요 비즈니스 메서드 (Methods)**
  - `markAsRead()`: 사용자가 알림을 확인하면 `readStatus`를 `true`로 전환합니다.
  - `createNotification(String recipient, String title, String content)`: 신규 도메인 이벤트 발생 시 알림 레코드를 인서트합니다.

### 2.7. DocumentAttachment (기안 상신 복수 참고/증빙 첨부파일 - Entity / VO)

**1. 구성 이유 및 역할**
  - **구성 이유:** 결재 기안 상신 시 본문 문서/도면 파일 외에 참고 증빙자료(견적서, 시방서, 사진 등)를 1:N 관계 복수로 자유롭게 첨부하고 무결성을 물리 통제하기 위함.
  - **역할:** 파일별 원본명, S3/MinIO 저장 키, 용량, SHA256 체크섬, 파일 무결성 상태(`fileState`) 보관 및 비동기 무결성 검증 큐 연동.

**2. 주요 속성 (Attributes)**
  - `attachmentId` (String / UUID): 첨부파일 고유 식별자 (PK).
  - `documentId` (String): 소속 부모 문서 마스터 ID (FK).
  - `attachmentRole` (AttachmentRole Enum): 첨부 역할 (`MAIN_DOCUMENT` 본문/도면, `REFERENCE_DOC` 참고 증빙자료, `ESTIMATE` 견적서).
  - `originalFileName` (String): 기안자가 업로드한 원본 파일명 (예: `2026_부품견적서.pdf`).
  - `storageKey` (String): S3/MinIO 물리 저장 객체 키 (`/attachments/2026/07/{uuid}.pdf`).
  - `fileSize` (Long): 파일 용량 (Byte).
  - `mimeType` (String): 파일 MIME 타입 (예: `application/pdf`, `image/png`).
  - `checksumSha256` (String): 백그라운드 무결성 검증용 SHA-256 해시값.
  - `fileState` (FileState Enum): 파일 무결성 상태 (`PENDING`, `VERIFYING`, `CONFIRMED`, `CORRUPTED`).

**3. 주요 비즈니스 메서드 (Methods)**
  - `confirmFile(String sha256)`: 비동기 파일 검사 큐 통과 시 `fileState`를 `CONFIRMED`로 바꾸고 SHA-256 해시를 확정 등록합니다.
  - `markAsCorrupted(String reason)`: 바이러스 감지나 체크섬 불일치 시 `fileState`를 `CORRUPTED`로 변경하고 사유를 남깁니다.

### 2.8. DocumentRelationMap (품목·프로젝트·문서 간 다차원 N:M 연관관계 매핑 - Entity)

**1. 구성 이유 및 역할**
  - **구성 이유:** 문서 마스터 또는 특정 첨부파일이 품목(`PART`), 프로젝트(`PROJECT`), 타 연관 문서(`DOCUMENT`), 설비(`EQUIPMENT`) 등과 맺는 N:M 유연한 연관관계를 중앙 교차 테이블로 추적하기 위함.
  - **역할:** 품목 클릭 시 연관된 모든 문서/첨부파일 역방향 조회의 지휘소 역할 및 변경 발생 시 파급 효과(Impact Analysis) 알림 대상 도출.

**2. 주요 속성 (Attributes)**
  - `relationId` (Long): 매핑 식별자 (PK).
  - `documentId` (String): 연관 문서 마스터 ID (FK).
  - `attachmentId` (String, Optional): 핀포인트 지정 시 연관 첨부파일 ID (FK).
  - `targetType` (TargetType Enum): 연관 대상 엔티티 구분 (`PART` 품목, `PROJECT` 프로젝트/기종, `DOCUMENT` 타결재문서, `EQUIPMENT` 설비).
  - `targetCode` (String): 연관 대상 식별 코드 (예: 품번 `PART-10023`, 프로젝트 `PROJ-EAGLE`).
  - `relationType` (RelationType Enum): 연관 성격 (`APPLIED_PART` 적용 품목, `REFERENCE_DOC` 참조 문서, `SUPERSEDED_DOC` 변경 대상 문서).
  - `createdBy` (String): 연관 매핑 등록자 ID.

**3. 주요 비즈니스 메서드 (Methods)**
  - `mapRelation(TargetType type, String code, RelationType relType)`: 문서/첨부파일과 외부 품목 및 프로젝트 간 연관관계를 신규 매핑 바인딩합니다.
  - `unmapRelation()`: 해제된 연관관계를 매핑 테이블에서 안전하게 제거합니다.
